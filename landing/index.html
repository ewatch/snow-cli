The safe gateway between LLMs and ServiceNow

snow-cli is the cleanest way to let coding agents and LLMs talk to a ServiceNow instance — machine-readable by design, least-privilege by default, and built to drive instance changes through pipelines like GitLab CI.

Why snow-cli

Built for the way agents actually work

ServiceNow's APIs are powerful but verbose. snow-cli gives an LLM a small, predictable, auditable surface — and gives you control over exactly what it's allowed to touch.

🤖

LLM-native output

A global --output flag renders results as JSON, JSON Lines, TOON, or CSV. Predictable, machine-readable shapes that agents can parse — no UI scraping, no guessing at HTML.

🔒

Least privilege first

A dedicated snow-cli-ro binary plus a fail-closed request policy — new commands are denied until explicitly audited. Pair it with a snc_read_only identity for defense in depth.

🚀

Pipeline-ready

Single static binary, credentials via env vars, deterministic exit codes and JSON errors. Drops straight into GitLab CI to promote instance changes safely.

One command, four shapes

Pick the output your tool actually wants

A global --output flag renders the same data as pretty JSON for humans, JSON Lines for streaming, TOON for token-efficient LLM context, or CSV for spreadsheets and pipelines.

$ snow-cli --output json table list incident --fields number,short_description,priority --limit 2
[ { "number": "INC0010023", "short_description": "Auth service degraded", "priority": "1" }, { "number": "INC0010024", "short_description": "Email digest delayed", "priority": "3" } ]
# one JSON object per line — ideal for streaming & log pipelines {"number":"INC0010023","short_description":"Auth service degraded","priority":"1"} {"number":"INC0010024","short_description":"Email digest delayed","priority":"3"}
# TOON — compact tabular shape that costs far fewer tokens in an LLM prompt [2]{number,short_description,priority}: INC0010023,Auth service degraded,1 INC0010024,Email digest delayed,3
number,short_description,priority INC0010023,Auth service degraded,1 INC0010024,Email digest delayed,3

Why TOON? It carries the same rows as JSON but drops repeated keys and punctuation, so an agent can fit more records into the same context window — fewer tokens, lower cost, same data.

Security by design

Give agents access — not the keys to the instance

Security isn't a flag you remember to set. snow-cli enforces read-only access at two layers and stores secrets in the OS keychain, so an over-eager agent simply can't write.

  • ✓Two-layer read-only policy. The command layer refuses mutating subcommands (table update, script run, api post); the request layer blocks any non-GET HTTP call.
  • ✓Fail-closed by default. Unaudited commands are denied until explicitly allowed — no silent escalation.
  • ✓Server-side defense in depth. Bind tokens to a snc_read_only user or a table_read OAuth scope so the identity itself can't write.
  • ✓Credentials in the OS keychain. OAuth 2.0 authorization-code with PKCE — no long-lived secrets sitting in plaintext config.
# Hand agents the locked-down binary + a read-only identity
$ snow-cli profile add agent-ro \
  --instance https://dev12345.service-now.com \
  --auth-method oauth2 \
  --oauth-grant-type authorization-code \
  --oauth-scope table_read

$ snow-cli-ro --profile agent-ro table update incident SYS_ID
error: command 'table update' is not permitted in read-only mode

$ snow-cli-ro --profile agent-ro api get /api/x_app/status
{ "status": "ok" }

Read-only mode ships as both the snow-cli-ro binary and a --read-only flag on the full CLI.

CI / CD

Adopt instance changes through your pipeline

Treat ServiceNow like the rest of your stack. Export, validate, and import records as portable artifacts — driven from GitLab CI with credentials supplied as masked variables.

# .gitlab-ci.yml — promote data between instances
promote:
  script:
    # credentials arrive as masked CI variables
    - export SNOW_CLI_PASSWORD="$SNOW_PW"
    - snow-cli --output json data export \
        sys_script_include --query 'sys_scope=x_app' \
        --out artifact.json
    - snow-cli data import --profile staging artifact.json

Portable artifacts

data export writes a single table to JSON you can diff, commit, and review in a merge request — then data import applies it downstream.

Move files between scopes

scope move-file relocates an application file into another custom scope without changing its sys_id — with --dry-run to preview every change first.

Review & discover

Inspect apps, read records, follow the relations

Whether you're reviewing a change or reverse-engineering an unfamiliar app, snow-cli turns the instance into something you can actually navigate from the terminal.

1

Inspect a scope

scope inspect x_my_app summarizes an application's metadata; scope inventory exports every artifact as normalized rows.

2

Read the records

table get / table list with field selection and encoded queries — and table schema to understand a table before you touch it.

3

Discover relations

codesearch search finds where a Script Include, API, or pattern is referenced — across business rules, widgets, and includes.

4

Review the change

Pipe machine-readable output into your review workflow, or hand it to an agent that explains what an update set actually does.

# Map an unfamiliar app, then trace a dependency
$ snow-cli scope inspect x_my_app --details full
x_my_app · 142 artifacts · 38 business rules · 17 script includes · 9 widgets

$ snow-cli --output csv scope inventory x_my_app
$ snow-cli codesearch search "new x_my_app.OrderService" --current-scope
✓ references found in 3 business rules and 1 scheduled job
