ServiceNow CLI for agents

Inspect ServiceNow workflows with your coding agent.

snow-cli gives developers and terminal-based agents a structured command surface for ServiceNow. Investigate Flow Designer artifacts, trace instance code, map custom applications, and review recent changes without navigating the UI for every question.

The agent workflow

Turn a Flow Designer question into inspectable evidence.

A developer asks Claude Code, opencode, pi, or another terminal harness to review a flow. The agent invokes snow-cli, reads the table schema before it queries Flow Designer records, traces related scripts, and reviews recent development activity after the developer publishes a new Flow Designer version.

Approval flow review

  1. Ground the queryRead the Flow Designer table schema first.
  2. Find the flowQuery Flow Designer records by scope and name.
  3. Trace dependenciesSearch scripts referenced by the implementation.
  4. Review the iterationRead recent Customer Updates by recency.

  --query 'ORDERBYDESCsys_updated_on' \
  --fields name,type,action,sys_updated_on,update_set --limit 25

Deliberate boundary: snow-cli exposes the records and related code an agent needs for inspection and review. It does not claim to replace the visual Flow Designer authoring canvas. The transcript shows commands only, with no simulated instance output.

USE CASE 01

ServiceNow code search

Find script references across the instance.

Use codesearch search when a flow, business rule, or Script Include points to logic you need to trace. Restrict the search to a source table, an application scope, or the current scope.

  • Search scripts, business rules, Script Includes, and other configured Studio search sources.
  • Return bounded, structured results for an agent to inspect.
search one code source
$ snow-cli --read-only --output json codesearch search GlideAjax \
  --source-table sys_script_include \
  --scope x_acme_ops --limit 50

Command surface: codesearch search <QUERY> --source-table --scope --limit

USE CASE 02

Custom application overview

Inventory the custom apps installed on an instance.

scope list reads scope, Store application, and plugin metadata and classifies results by origin. Filter to custom applications to identify the app scopes relevant to a review.

  • Separate custom applications from platform, Store, and plugin entries.
  • Return structured scope metadata for a focused application overview.
map custom application scopes
$ snow-cli --read-only --output json scope list \
  --kind custom-app

Command surface: scope list --kind custom-app

USE CASE 03

Recent development activity

Review the latest customer updates by recency.

Query the Customer Updates table with a descending encoded order and an explicit record limit. The result gives an agent a bounded view of recently updated application records without inventing an activity metric.

  • Select only the fields needed for review.
  • Use encoded queries to narrow by update set, type, scope, or time window when needed.
latest customer update records
$ snow-cli --read-only --output json table list sys_update_xml \
  --query 'ORDERBYDESCsys_updated_on' \
  --fields name,type,action,sys_updated_on,update_set \
  --limit 25

Command surface: table list <TABLE> --query --fields --limit

A bounded agent interface

Structured enough for tools. Legible enough for developers.

snow-cli uses the same command surface in an interactive terminal, an agent harness, or a shell pipeline. Instance permissions still determine which ServiceNow records the authenticated identity may read.

01

Read-only execution policy

--read-only allows the inspection commands shown here while denying table mutations and non-GET raw API requests.

02

Explicit result bounds

table list defaults to a bounded result and supports explicit --limit, field selection, and encoded queries.

03

Machine-readable output

The global --output option supports JSON, JSON Lines, CSV, TOON, text, and automatic format selection.

Start with a read-only review

Bring ServiceNow evidence into the terminal.

After installing a release, add a profile for your instance and authenticate. Use an identity whose ServiceNow roles match the access you intend to grant.

profile setup
$ snow-cli profile add dev \
  --instance https://example.service-now.com
$ snow-cli auth login --profile dev
